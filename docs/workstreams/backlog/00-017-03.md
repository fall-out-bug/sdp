# 00-017-03: pi model review and synthesis

## Goal

Use `pi` as the model runner for GLM/Kimi review and produce a synthesized SDP verdict input.

## Acceptance Criteria

- [ ] AC1: GLM and Kimi are addressed through provider aliases from env/config, not hard-coded model ids.
- [ ] AC2: OpenRouter is used only as fallback when a primary provider is unavailable.
- [ ] AC3: Provider failures are recorded in run telemetry with provider, model alias, status, latency, and error.
- [ ] AC4: The synthesizer maps findings to `P0`-`P3`, marks `P0`/`P1` blocking, and requires evidence to dismiss plausible blocking findings.
- [ ] AC5: Contradictory model results produce `CHANGES_REQUESTED` or `ESCALATED` with a reason, not silent approval.

## Context

The model layer is deliberately behind `pi`. SDP owns the protocol and artifacts; `pi` owns model invocation and provider env.

## Dependencies

- 00-017-01: schema contract.
- 00-017-02: context packet and test evidence.

## Steps

1. Define the internal interface for invoking `pi` with a context packet and model alias.
2. Add GLM and Kimi reviewer calls using the alias interface.
3. Add OpenRouter fallback routing without changing the public verdict contract.
4. Implement synthesis over model findings, deterministic test evidence, and provider errors.
5. Add fake-runner tests for clean review, blocking finding, provider failure, and GLM/Kimi contradiction.

## Expected Result

The gate can ask independent GLM/Kimi reviewers for findings and produce a deterministic synthesis result from their outputs.

## Constraints

- Do not store provider secrets in SDP.
- Do not couple reviewer roles to concrete model version strings.
- Do not treat model silence as approval.
